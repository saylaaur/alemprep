## 0. Рамка проекта (читать первым)

Это **MVP под друга-партнёра**. Разделение труда:

- **Наша зона (Жангирхан + Claude):** построить рабочий, красивый MVP — код, дизайн,
  автоматический контент-пайплайн, деплой.
- **Зона партнёра:** деньги на API/инфраструктуру, продвижение, дальнейшее наполнение и развитие.

Из этого следует главное: **не строим контент-империю**. Цель — продукт, который убедительно
работает и хорошо выглядит на 3 предметах, с достаточным банком задач для реального
использования. Дальше масштабирует партнёр.

Ограничения от заказчика:

1. **Ноль ручной возни с SQL.** Скрипт пишет в базу сам. Человек запускает команду и
   подключает ключи.
2. **Деньги на API есть.** Уходим с Gemini на **Claude Haiku**.
3. **Заказчик сам тестирует.** ИИ-верификатор не нужен — проверку делает человек через
   review-страницу. Оставляем только дешёвые детерминированные проверки.

## 1. Зафиксированные технические решения

| Вопрос | Решение | Почему |
|---|---|---|
| AI для контента | **Claude Haiku 4.5** (Anthropic SDK) | Есть бюджет; качество стабильнее free-tier Gemini; SDK уже в зависимостях |
| Бэкенд | **Supabase — оставляем** | Уже подключён (OAuth, RLS, миграции); смена = недели ради нуля пользы. «Напрягал» не он, а ручной SQL — его убираем |
| Заливка контента | **Прямая запись в БД** через service-role ключ, `is_published=false` | Убирает ручную вставку SQL |
| Контроль качества | **Ручной тест заказчиком** + детерминированные проверки (zod, дедуп, санити LaTeX) | ИИ-верификатор не нужен — человек проверяет в review-странице |
| Старые данные Gemini | БД-задачи (24 шт.) **оставляем**; Gemini-код и `@google/genai` удаляем; JSON-выгрузки в архив | Рабочий контент не трогаем, чистим только код |
| Дизайн | **Сначала ревью текущего, потом решение** | Визуал уже прокачан; редизайн с нуля — риск и время |
| Скоуп MVP | **Математика + Информатика + Физика** | Подтверждено заказчиком |

## 3. Решение по картинкам (порядок без двойной работы)

Ключевой вопрос: сначала поддержка картинок или сначала генерация?

1. **Математика + информатика** (~90% текст) — генерим **сейчас**, текстовым пайплайном.
   Картинок не ждём, основной объём закрываем быстро.
2. **T11 (поддержка картинок)** — делаем **после** math/информатики, но **до** физики.
3. **Физика** — генерим **одним проходом уже с картинками**. Так не теряем ~40% задач и
   не перегоняем заново.

**Нюанс генерации картиночных задач:** ответ завязан на график/схему, которую нельзя
перерисовать → менять числа нельзя. Для таких задач пайплайн делает **1–2 переформулировки
с той же картинкой** (не 5 вариантов) и грузит изображение в Supabase Storage. Контекстные
задания: картинка крепится к строке `contexts` (общий стимул), подвопросы её наследуют.

Итоговый порядок: **пайплайн → math + информатика → T11 → физика**.

## 4. Архитектура автопайплайна

```
[PNG пробника]
     │  Haiku vision
     ▼
[transcribe] ── reference JSON {topic_slug, type, body, correct, explanation}
     │            графики → если нет T11: skip; если есть: грузим картинку в Storage
     ▼  Haiku
[generate]  ── N вариантов (меняем числа/контекст; есть картинка → 1–2 варианта)
     │
     ▼  детерминированные проверки (НЕ ИИ)
[validate]  ── zod-схема + дедуп (одинаковые числа) + санити LaTeX
     │
     ▼  service-role, прямая запись
[insert]    ── questions, is_published = false
     │
     ▼
[review-страница] ── заказчик смотрит, тестирует, публикует пачкой (без SQL)
```

Человек участвует в: (1) запуске команды, (2) ключах, (3) проверке/публикации в review-UI.

## 5. План по эпикам

**Эпик A — Автопайплайн на Haiku** ⭐
A1 transcribe на Anthropic SDK · A2 generate на Haiku · A3 детерминированные проверки
(zod+дедуп+LaTeX) · A4 прямая запись в БД через service-role (`is_published=false`) ·
A5 команда-оркестратор `gen:all` · A6 лог стоимости · A7 чистка Gemini-кода и зависимости.

**Эпик B — Темы + наполнение (текст)**
B1 миграция тем информатики · B2 миграция тем физики · B3 прогон math (→~250) ·
B4 прогон информатики (→~150).

**Эпик C — Картинки + физика**
C1 T11 (поле `image?` в `body` и `contexts`, Storage-загрузка, рендер) ·
C2 прогон физики с картинками (→~150).

**Эпик D — Review-страница**
D1 простая страница `/admin/review` (или тоггл) для просмотра и публикации пачкой.

**Эпик E — Ядро продукта**
E1 T5 мобильная адаптация · E2 T7 полный пробник.

**Эпик F — Дизайн**
F1 скриншоты текущего · F2 решение полировка/редизайн (Google Stitch для идей, точечно).

**Эпик G — Деплой**
G1 Vercel + env · G2 домен/`*.vercel.app` · G3 OAuth+Supabase redirect · G4 smoke-тест мобилки.

## 6. Что нужно от тебя (входы)

| Что | Зачем | Когда |
|---|---|---|
| `ANTHROPIC_API_KEY` в `.env.local` | Haiku-пайплайн | Перед Эпиком A |
| `SUPABASE_SERVICE_ROLE_KEY` в `.env.local` | Прямая запись в БД | Перед A4 |
| PNG-материалы (math/информатика/физика) | Исходники | Перед B/C |
| Домен или ок на `*.vercel.app` | Деплой | Эпик G |

⚠️ Anthropic API — **отдельный платный счёт**, не подписка Claude Pro/Max. Стоимость всего
контента MVP — **единицы долларов**.

## 7. Бюджет и сроки

**Деньги:** домен ~$10 + Anthropic API ~$5–15 + остальное free tier = **~$15–25** на MVP.

**Время:** при подписке Max + Claude Code субагентах + личном тесте заказчика — **~1 неделя**
на код. Узкое место не код (он параллелится агентами), а сбор исходников и ручная проверка —
это серийное человеческое время.

## 8. Риски

| Риск | Митигация |
|---|---|
| Без ИИ-верификатора пройдёт кривая задача | Ручной тест заказчика + дедуп/zod/LaTeX-санити; review перед публикацией |
| Физика без картинок теряет ~40% | T11 до прогона физики (см. §3) |
| Сбор исходников затянет неделю | Подготовить PNG заранее, до старта спринта |
| Редизайн съест время | Сначала ревью; редизайн только если реально плохо |
