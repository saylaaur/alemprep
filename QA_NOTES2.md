# QA перед релизом — заметки

Ветка `qa/pre-launch` от `main` (72b9fce). Не мержится, не пушится.
Формат: находка → фикс → `/verify` → коммит. Один коммит = один логический фикс.

## Область проверки
Лендинг · вход · дашборд · тренажёр (3 типа вопросов) · пробник + калькулятор ·
прогресс · admin/review · settings.
Сквозные темы: геймификация на границах + идемпотентность, паритет ru/kk,
a11y (фокус/aria/клавиатура), запросы дашборда (N+1).

## Важное отклонение от ТЗ
`main` **не включает** ветку `feature/finish` (полноценные settings, стрик
из пробника, восстановление пробника, focus-trap в drawer) — она отдельно и
не смёржена. `/settings` на `main` — заглушка (`comingSoon`). QA идёт по
`main` как договорено («от main»); заглушка settings отмечена как отдельный
пункт, а не разбирается как «сломанное состояние» полноценной фичи.

## Находки

### 1. Калькулятор пробника «зависает» после результата в экспоненциальной записи — CONFIRMED, фикс
`components/practice/Calculator.tsx` + `lib/calculator.ts`. `formatValue()` использует `String(v)`.
Для чисел ≥1e21 или очень маленьких (например физическая константа 1.6×10⁻¹⁹, заряд электрона —
вполне реалистичный ввод на экзамене по физике) JS даёт экспоненциальную запись `"1.6e-19"`.
Дальше:
- `append()` (нажатия на кнопки цифр) ничего не валидирует и просто дописывает к строке →
  выражение необратимо содержит `e`;
- `evaluate()`/`tokenize()` не понимает символ `e` → любое повторное `=` даёт «Ошибка»;
- `ALLOWED_INPUT` регэксп на инпуте тоже не разрешает `e`, так что после появления `e` в
  дисплее любой ручной ввод символа блокируется (тестируется на всей строке, а не диффе).
Итог: калькулятор становится непригоден до нажатия Clear — на экзамене по физике это боль.
**Фикс:** `formatValue` разворачивает экспоненциальную запись в обычную десятичную строку.

### 2. Гонка при начислении XP за два блока пробника + несогласованное состояние при частичном сбое — CONFIRMED, фикс
`components/practice/MockExamView.tsx` (`handleSubmit`) + `lib/supabase/practice-actions.ts`
(`finishExamSession`, `recordAttempt`) + `components/practice/PracticeView.tsx` (`check`).

**Гонка XP.** Два блока пробника (математика + второй предмет) отправлялись одновременно через
`Promise.all`. `finishExamSession` начисляет XP паттерном read-then-write (`SELECT xp` →
`UPDATE xp = old + gain`) без атомарности. При двух параллельных вызовах на одном `profiles.id`
второй `UPDATE` перезаписывает результат первого — XP одного из блоков пробника **молча
теряется**. Не гипотетика: `Promise.all` — гарантированно параллельный старт.
Фикс: (а) в `handleSubmit` — последовательная отправка блоков вместо `Promise.all`;
(б) в `finishExamSession` — `UPDATE ... WHERE finished_at IS NULL` с проверкой числа затронутых
строк вместо read-then-check-then-write, чтобы закрыть более узкое TOCTOU-окно между начальной
проверкой идемпотентности и записью результата (двойной клик/ретрай сети на границе гонки).
Покрыто тестом с настоящим `Promise.all([finishExamSession(input), finishExamSession(input)])`
на одной сессии — второй вызов не дублирует XP/попытки.

**Несогласованное состояние при частичном сбое.** Если после успешной записи сессии/попыток
падал следующий шаг (вставка попыток или обновление профиля), БД оставалась в противоречивом
состоянии: например, попытка вставлена, а XP/стрик — нет (или наоборот), без пути назад.
Аналогично в `recordAttempt` (тренажёр): если `attempts.insert` проходил, а обновление
`profiles` падало, попытка оставалась в БД без начисленного XP/стрика — и без отката клиент не
мог понять, что что-то пошло не так. Фикс: обе server actions теперь делают компенсирующий
откат при сбое промежуточного шага (удаляют уже вставленную попытку / возвращают сессию к
`finished_at: null`), возвращая явную ошибку, чтобы клиент мог безопасно повторить отправку без
дублирования. `PracticeView.check()` также перестал выставлять `revealed=true` синхронно ДО
ответа `recordAttempt` — теперь верификация показывается только после успешного сохранения;
при сетевой ошибке — баннер `saveError` и кнопка «Проверить» остаётся активной для повтора
(раньше ответ считался проверенным в UI, даже если сохранение в БД тихо падало).

### 3. Фокус-ловушка отсутствует в мобильном drawer — CONFIRMED, фикс (a11y)
`components/layout/MobileNav.tsx`. При открытии drawer фокус не уводится внутрь, Tab свободно
уходит на контент под оверлеем (`aria-modal="true"`, но без реального trap — противоречие для
скринридеров), Escape закрывает, но фокус после закрытия не возвращается на кнопку-бургер.
**Фикс:** focus-trap (цикл Tab/Shift+Tab внутри панели), `inert` на закрытой панели, возврат
фокуса на триггер при закрытии.

### 4. ARIA-семантика вариантов ответа (тренажёр + пробник) — CONFIRMED, фикс (a11y, low-risk)
`components/practice/PracticeView.tsx` (`SingleAnswer`/`MultiAnswer`) и
`components/practice/MockExamView.tsx` (варианты single/multi). Варианты — это `<button
aria-pressed>`, а по факту взаимоисключающий выбор (single) или группа чекбоксов (multi).
`aria-pressed` на кнопках не сообщает скринридеру «выбери один из» — каждая кнопка объявляется
независимым тоглом. **Фикс:** добавлена `role="radiogroup"`/`role="radio"`+`aria-checked` для
single и `role="group"`+`role="checkbox"`+`aria-checked` для multi — только ARIA-атрибуты,
логика клика/клавиатуры не тронута (низкий риск регресса).

### 5. Проверено, проблем не найдено
- **N+1 в запросах дашборда/прогресса** (`lib/supabase/queries.ts`): `getGamification`,
  `getProgressData`, `getSubjectsWithCounts`, `getUnpublishedQuestions`, `getPairExamBlocks` —
  везде батчинг через `Promise.all` + одиночные `.in()`-запросы, ни одного query-в-цикле.
- **Границы геймификации** (`lib/gamification.ts`, `lib/streak.ts`): уровни/пороги/стрик уже
  плотно покрыты тестами на границах (0/99/100/299/300…, mastery 9/10 vs 10/10, exam-90 на
  0.89/0.90, будущая `lastActiveDate` при рассинхроне часов). Пересчитывать не стал —
  тесты валидны и проходят.
- **Идемпотентность `finishExamSession` при повторном (не гоночном) вызове** — уже была
  защищена веткой `prior.finished_at` и покрыта тестом; фикс №2 усиливает именно гоночный кейс.
- **RLS/авторизация admin-действий** (`lib/supabase/admin-actions.ts`): `publishQuestion` /
  `deleteQuestion` / `publishAll` — каждый проверяет `requireAdmin()` на сервере независимо от
  UI-гейта в `admin/review/page.tsx`. Нельзя обойти через прямой вызов server action.
- **Паритет ru/kk**: 224/224 ключей (скрипт-сверка), namespace'ы `t()`/`useTranslations()` в
  коде совпадают 1:1 со всеми 15 top-level namespace'ами в messages/*.json.
- **Плейсхолдеры ICU plural ru vs kk различаются по числу категорий** (ru: `one/few/other`,
  kk: только `other`) — это НЕ баг: у казахского в CLDR только одна категория множественного
  числа, `other`-only — грамматически корректная локализация, не пропущенный перевод.

### 6. Не баг, но на заметку (см. «Открытые вопросы»)
- 14 строк, где ru===kk дословно (`AlemPrep`, `Математика`, `Прогресс`, `Навигатор`,
  `Калькулятор` и т.п.) — похоже на устоявшиеся заимствования в разговорном казахском, но я не
  берусь подтвердить это как носитель; листинг — в «Открытые вопросы».
- `MathText` рендерит формулы через KaTeX с CDN (`cdn.jsdelivr.net`) без бандлинга и без
  таймаута ретраев: если CDN недоступен (блокировщик/сеть), retry-цикл `setTimeout(100)`
  крутится, пока компонент не размонтируется, а формулы весь это время показываются как сырой
  LaTeX-текст. Архитектурный риск (единая точка отказа), не «баг» в рамках текущего PR.
- `/settings` на `main` — заглушка (`comingSoon`), полноценная реализация в отдельной
  нессмерженной ветке `feature/finish`. Не разбирался как «сломанное состояние».
